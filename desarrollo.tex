\chapter{Desarrollo}
\section{Metodología}
Para el desarrollo de este proyecto se ha optado por una metodología iterativa incremental, siguiendo los principios de desarrollo ágil y permitiendo la adaptación del desarrollo a las necesidades emergentes. Esta metodología se caracteriza por definir un número ``N'' de iteraciones en las que se va entregando por cada una de ellas una versión más completa del proyecto~\cite{larman2004applying}.

Asimismo, no se consideró correcta la elección de otras metodologías ágiles como ``\gls{scrum}'' ya que a pesar de ser un trabajo colaborativo, la línea que se desarrolla en este tfg era independiente.

La \gls{iterinc} encaja especialmente bien con el problema de un modelo de
datos en evolución: cada incremento entrega una versión funcional del
framework y, al cerrarlo, se analiza el estado del proyecto y las necesidades
emergentes para planificar el siguiente. De este modo, la propia metodología
absorbe la incertidumbre que motiva el trabajo.

\subsubsection{Diseño e implementación del modelado de datos}
Con el entorno reproducible en marcha, el Incremento~1 se centró en consolidar
la capa de persistencia. El backend organiza los modelos ORM por dominio
(\texttt{core}, \texttt{sysadmin}, \texttt{annotations},
\texttt{hierarchies}, \texttt{species}, \texttt{files},
\texttt{geographics}, \texttt{screens}). La pieza central es
\texttt{FunctionalObject}: una clase base de la que heredan las entidades de
dominio, que aporta identificador, \texttt{uuid}, propiedad (\emph{ownership}),
marcas de tiempo y atributos extensibles. Sobre ella se apoya un sistema de
tipos de objeto (\texttt{ObjectType}) que asigna a cada clase un código
numérico, lo que habilita el tratamiento polimórfico de las entidades.

El subsistema de autenticación y autorización emplea herencia polimórfica:
\texttt{Identity} y \texttt{Role} heredan de \texttt{Authorizable}, de modo
que las columnas comunes (\texttt{name}, \texttt{uuid}) residen en la tabla
padre y solo las específicas (\texttt{email}, \texttt{can\_login}, \ldots) en
la tabla hija. El modelo de permisos se completa con \texttt{ACL},
\texttt{ACLExpression} y \texttt{ACLDetail}, junto con organizaciones, grupos
y roles. El prefijo de tablas es configurable mediante
\texttt{get\_table\_prefix()} (\texttt{ub\_} por defecto), lo que evita
colisiones cuando el framework convive con otras tablas en la misma base de
datos.

\subsubsection{Versionado e histórico con SQLAlchemy-Continuum}
Para cumplir el requisito de histórico (RF-10) se integró
\gls{continuum}. La inicialización es delicada: \texttt{make\_versioned} debe
invocarse \emph{antes} de definir cualquier modelo o crear la base ORM, por lo
que se ubicó en el módulo \texttt{initializer}. Se habilitó el versionado
nativo (\texttt{native\_versioning}) y se configuró Alembic para soportarlo,
de modo que las entidades núcleo (\texttt{FunctionalObject},
\texttt{Authorizable}, \texttt{Identity}, \texttt{FileSystemObject}, \ldots)
mantienen automáticamente sus tablas de versiones. Las migraciones de esquema
se gestionan con Alembic, con un conjunto de \emph{revisions} versionadas en
el repositorio.

\subsubsection{Punto de entrada e inicialización}
El framework expone una única función \texttt{initialize(config)} que
normaliza la configuración con Pydantic Settings, prepara la base ORM y el
gestor de sesiones, crea la aplicación FastAPI e inicializa de forma opcional
Redis, el \emph{worker} Celery y Firebase. Esta función es el contrato de uso
del backend como librería: un proyecto consumidor solo necesita un diccionario
de configuración y sus propios modelos, sin tocar el núcleo.

\subsubsection{Decisiones de diseño y problemas encontrados}
El problema más delicado del Incremento~1 fue el \textbf{orden de
inicialización de \gls{continuum}}. La extensión instrumenta los modelos en el
momento de su definición, por lo que \texttt{make\_versioned} debe ejecutarse
antes de importar cualquier modelo o crear la base declarativa; ubicarlo en el
módulo \texttt{initializer}, fuera de los modelos, resolvió un fallo
intermitente de mapeo que solo se reproducía según el orden de importación. La
segunda decisión relevante fue el \textbf{prefijo de tablas configurable}: sin
él, el framework colisionaría con tablas existentes al integrarse en un
proyecto con su propia base de datos; resolverlo mediante
\texttt{get\_table\_prefix()} mantiene la genericidad (RNF-01). Por último, la
\textbf{herencia polimórfica} de \texttt{Authorizable} obligó a documentar que
las consultas de identidades y roles requieren \emph{join} con la tabla padre,
algo no evidente que se trasladó a la documentación interna del proyecto.

\subsection{Iteración 2: contrato de comunicación back--front}
El Incremento~2 resuelve el acoplamiento histórico entre frontend y backend
mediante un contrato uniforme, documentado en \texttt{docs/CONTRACT.md}.

\subsubsection{Envoltorio universal de respuesta}
Toda respuesta de la API --- de éxito o de error --- adopta la misma forma:
un objeto con \texttt{content}, \texttt{count} e \texttt{issues}. El modelo se
implementa con Pydantic (Algoritmo~\ref{lst:envelope}).

\begin{lstlisting}[language=Python, caption={Envoltorio de respuesta normalizado (\texttt{responses.py})}, label={lst:envelope}]
class ResponseEnvelope(BaseModel):
    content: Optional[Any] = None
    count: int = 0
    issues: List[Issue] = Field(default_factory=list)

    @classmethod
    def ok(cls, content=None, count=None, issues=None):
        if count is None:
            count = len(content) if isinstance(content, list) \
                    else (1 if content is not None else 0)
        return cls(content=content, count=count, issues=issues or [])

    @classmethod
    def fail(cls, message, code=None, location=None):
        return cls(content=None, count=0,
                   issues=[Issue.error(message, code=code,
                                       location=location)])
\end{lstlisting}

El Algoritmo~\ref{lst:envjson} ilustra la forma concreta del envoltorio en un
caso de éxito y en un caso de error de validación.

\begin{lstlisting}[style=docker, caption={Envoltorio en éxito (arriba) y en error de validación (abajo)}, label={lst:envjson}]

\subsubsection{Descubrimiento}
Dos endpoints de descubrimiento completan el contrato:
\texttt{GET /api/sys/entities} lista todas las entidades CRUD detectadas en
las rutas de la aplicación junto con sus capacidades
(\emph{list/get/create/update/delete/schema}), y
\texttt{GET /api/sys/discovery} lista los servicios externos detectados a
través de Traefik. El descubrimiento es la pieza que hace el sistema
\emph{autodescriptivo}: el frontend puede preguntar al backend qué entidades
existen en lugar de tenerlas codificadas.

\subsubsection{Capa cliente genérica en ngt-gui}
En el frontend se implementó una capa cliente reutilizable, exportada desde
\texttt{ngt-gui}. \texttt{EntityClient} ofrece un cliente tipado por
\emph{path} de entidad; \texttt{SchemaService} cachea el esquema por entidad y
lanza error si el envoltorio trae incidencias de tipo \texttt{ERROR};
\texttt{DiscoveryService} expone la lista de entidades como observable. Un
\texttt{ApiErrorInterceptor} opcional convierte cualquier error HTTP en un
\texttt{ApiResponse<null>} con \texttt{issues}; se diseñó \emph{opt-in} para
no romper el código heredado (RNF-06).

\begin{lstlisting}[language=Java, caption={Uso de la capa cliente genérica desde un componente Angular}, label={lst:entityclient}]
const roles = this.api.for<Role>('/roles');
roles.list().subscribe(res => /* ApiResponse<Role[]> */);
roles.create({ name: 'editor' }).subscribe(res => ...);
roles.schema<JSONSchema>().subscribe(res => ...);
\end{lstlisting}

Junto a \texttt{EntityClient}, la capa incluye \texttt{SchemaService} (caché
de esquemas por entidad y por \emph{bundle}) y \texttt{DiscoveryService}
(lista observable de entidades), como muestra el Algoritmo~\ref{lst:services}.

\begin{lstlisting}[language=Java, caption={SchemaService y DiscoveryService}, label={lst:services}]
this.schemas.get('/roles').subscribe(s => /* JSON Schema */);
this.schemas.invalidate('/roles');          // forzar recarga
this.schemas.getBundle().subscribe(b => /* SchemaBundle */);

this.discovery.loadEntities().subscribe(l => /* EntityDescriptor[] */);
this.discovery.entities$.subscribe(l => /* observable */);
this.discovery.findEntity('roles');
\end{lstlisting}

Es importante destacar una decisión de compatibilidad: el
\texttt{BackendService} heredado, con sus \textasciitilde272 métodos
\emph{hardcoded}, sigue funcionando igual que antes; la nueva capa convive con
él y es la opción recomendada para entidades nuevas. La migración progresiva
del servicio heredado queda fuera del alcance del Incremento~2.

\subsubsection{Decisiones de diseño y problemas encontrados}
La principal tensión del Incremento~2 fue la \textbf{compatibilidad hacia
atrás}. El \texttt{BackendService} heredado estaba en producción y cualquier
cambio en la forma de las respuestas lo habría roto. La solución fue diseñar
la nueva capa como estrictamente \emph{aditiva}: el envoltorio se introduce en
los endpoints nuevos y en los manejadores de error globales, mientras que el
servicio antiguo sigue devolviendo el cuerpo crudo. El \texttt{ApiErrorInterceptor}
se dejó \emph{opt-in} por el mismo motivo: registrarlo globalmente habría
alterado el flujo de error de todo el código existente que captura
\texttt{error: (err) => \ldots}. Otra decisión fue \textbf{normalizar también
los errores}, no solo los éxitos: capturar \texttt{HTTPException},
\texttt{RequestValidationError} y \texttt{Exception} en manejadores globales
garantiza que el frontend jamás reciba dos formas distintas de fallo, lo que
simplifica radicalmente el manejo de errores en el cliente.

\subsection{Iteración 3: generación dinámica de formularios con Formly}
El Incremento~3 es el núcleo de la aportación y está documentado en
\texttt{docs/DYNAMIC\_FORMS.md}. La meta es obtener \emph{un formulario por
entidad sin que nadie escriba HTML específico}, partiendo del esquema que el
Incremento~2 ya publica.

\subsubsection{Enriquecimiento del esquema en el backend}
La conversión \texttt{sqlalchemy\_to\_pydantic} se amplió para inyectar, por
columna, pistas de presentación en el \texttt{json\_schema\_extra} mediante la
función \texttt{\_ui\_hints\_for\_column}. Estas pistas son heurísticas
derivadas del tipo SQLAlchemy, resumidas en la tabla~\ref{tab:hints}.

\begin{table}[!htbp]
\caption{Heurísticas de enriquecimiento del esquema por tipo de columna.}
\label{tab:hints}
\centering
\begin{tabular}{|l|p{7.5cm}|}
\hline
\textbf{Columna SQLAlchemy} & \textbf{Pista resultante} \\ \hline
\texttt{Boolean} & \texttt{x-ui-widget: checkbox} \\ \hline
\texttt{DateTime} & \texttt{x-ui-widget: datepicker} (con hora) \\ \hline

El diseño respeta cualquier \emph{override} manual: lo que el desarrollador
declare en \texttt{column.info["ui"]} se aplana con prefijo \texttt{x-} y tiene
prioridad sobre la heurística. Así, la automatización cubre el caso común sin
impedir el ajuste fino. El Algoritmo~\ref{lst:hints} muestra el extracto real
de la función que implementa estas reglas.

\begin{lstlisting}[language=Python, caption={Heurística de pistas de UI por columna (\texttt{utils/common.py})}, label={lst:hints}]
def _ui_hints_for_column(column) -> Dict[str, Any]:
    hints: Dict[str, Any] = {}
    col_type = column.type
    type_str = str(col_type).upper()

    if isinstance(col_type, sqltypes.Boolean):
        hints["x-ui-widget"] = "checkbox"
    elif isinstance(col_type, sqltypes.DateTime):
        hints["x-ui-widget"] = "datepicker"
        hints["x-formly-props"] = {"showTime": True}
    elif isinstance(col_type, sqltypes.Enum):
        hints["x-ui-widget"] = "select"
    elif isinstance(col_type, sqltypes.String):
        length = getattr(col_type, "length", None) or 0
        if length and length > 500:
            hints["x-ui-widget"] = "textarea"
    elif "JSON" in type_str:
        hints["x-ui-widget"] = "json"

    if column.foreign_keys:
        fk = next(iter(column.foreign_keys))
        ref_table = fk.column.table.name
        hints["x-ui-widget"] = "select-lazy-loading"
        hints["x-options-endpoint"] = f"/{ref_table}/"
        hints["x-options-label"] = "name"
        hints["x-options-value"] = fk.column.name or "id"

    # Overrides manuales: column.info["ui"] y claves x-*
    info = getattr(column, "info", None) or {}
    ui_info = info.get("ui") if isinstance(info, dict) else None
    if isinstance(ui_info, dict):
        for k, v in ui_info.items():
            key = k if k.startswith("x-") else f"x-{k}"
            hints[key] = v
    return hints
\end{lstlisting}

\subsubsection{Decisiones de diseño y problemas encontrados}
La generación dinámica planteó varios retos cuya resolución conviene
documentar. El primero fue \textbf{dónde inyectar las pistas}: situarlas en el
\texttt{json\_schema\_extra} de Pydantic, y no en un canal paralelo, garantiza
que viajen \emph{dentro} del propio esquema y sobrevivan a cualquier
serialización o cacheo. El segundo fue la \textbf{precedencia}: la heurística
debía ser un valor por defecto, nunca una imposición, de ahí que los
\emph{overrides} de \texttt{column.info} se apliquen al final y ganen siempre.
El tercero fue la \textbf{resolución de \texttt{\$ref}}: Pydantic emite
referencias internas (\texttt{\#/\$defs/X}) para objetos anidados, lo que
obligó a que el conversor del frontend las resolviese de forma recursiva antes
de mapear tipos. Por último, se decidió \textbf{no soportar todavía}
\texttt{oneOf/anyOf/allOf}: Pydantic los emite en escenarios de unión que el
modelo actual no necesita, y resolverlos correctamente exige el discriminador
polimórfico que se aborda en el Incremento~4. Acotar este límite de forma
explícita evitó introducir complejidad especulativa.

\subsubsection{Conversor de esquema a Formly en el frontend}
La pieza clave del frontend es \texttt{schemaToFormly}, un conversor puro (sin
dependencia de UI) que traduce un \gls{jsonschema} a un arreglo
\texttt{FormlyFieldConfig[]}. Mapea los tipos primitivos a tipos Formly,
resuelve referencias internas (\texttt{\$ref}) de forma recursiva, traduce
objetos anidados a \texttt{fieldGroup} y arreglos de objetos a \texttt{repeat},
e interpreta las pistas \texttt{x-*}: \texttt{x-formly-type},
\texttt{x-ui-widget}, \texttt{x-options-endpoint}, \texttt{x-placeholder},
\texttt{x-hide-when}, \texttt{x-required-when} y \texttt{x-disabled-when} se
traducen a tipos, opciones y expresiones Formly. Además genera los validadores
de Angular a partir de \texttt{minLength}, \texttt{maxLength},

\subsubsection{Componente de formulario dinámico}
El componente \texttt{<ngt-dynamic-form>} ofrece tres modos de uso, en orden
creciente de dinamismo: a partir de un \texttt{FormlyFieldConfig[]} ya
construido, a partir de un esquema en bruto, o --- el caso más potente --- a
partir únicamente del \emph{path} de la entidad, resolviendo el esquema con
\texttt{SchemaService}.

\begin{lstlisting}[language=Java, caption={Formulario generado solo a partir del path de la entidad}, label={lst:dynform}]
<ngt-dynamic-form entityPath="/roles"
                  [(model)]="model"
                  (submitted)="save($event)">
</ngt-dynamic-form>
\end{lstlisting}

El componente admite modos \texttt{create}, \texttt{edit} y \texttt{view}
(este último fuerza solo lectura), \emph{overrides} por campo e

\subsection{Iteración 4: generación automática de esquemas}
El Incremento~4 formaliza la extracción de esquemas desde el ORM y los publica
en un \emph{bundle} único versionado, documentado en
\texttt{docs/SCHEMA\_GENERATION.md}. La iteración quedó \textbf{cerrada}: el
registro de entidades, el endpoint \texttt{/api/sys/schemas}, el versionado por
ETag, el reflejo del polimorfismo y de las relaciones del ORM, y sus pruebas
están en su sitio.

Cada llamada a una fábrica CRUD registra la entidad en un \emph{registry}
global (\texttt{EntityRegistration}: nombre, \emph{path}, factoría, clase ORM,
esquema Pydantic, etiquetas). El Algoritmo~\ref{lst:registry} muestra la estructura de registro.

\begin{lstlisting}[language=Python, caption={Registro de entidad (\texttt{schema\_registry.py})}, label={lst:registry}]
@dataclass
class EntityRegistration:
    name: str               # "roles"
    path: str               # "/roles"
    factory: str            # "simple" | "crudie"
    orm_class: Type | None
    pydantic_schema: Type[BaseModel] | None
    tags: list[str]
\end{lstlisting}

A partir de él, \texttt{build\_schema\_bundle}
construye un documento con el esquema enriquecido de todas las entidades y
\texttt{compute\_etag} calcula un hash SHA-256 truncado, encapsulado como
validador débil \texttt{W/"\ldots"}. El Algoritmo~\ref{lst:bundle} muestra la
forma del \emph{bundle}.

\begin{lstlisting}[style=docker, caption={Documento del bundle de esquemas}, label={lst:bundle}]
{ "$schema": "https://json-schema.org/draft/2020-12/schema",
  "version": "9f3a1b2c4d5e6f70",
  "generated_at": "2026-05-11T10:38:02Z",
  "entities": {
    "roles":      { "type": "object", "properties": { } },
    "identities": { "type": "object", "properties": { } }
  } }
\end{lstlisting} El bundle incluye \texttt{version} (el
ETag, estable entre llamadas con la misma BD/ORM) y \texttt{generated\_at}
(metadato que no participa en el hash). Tanto el \emph{bundle} como cada
\texttt{/<entity>/schema.json} responden \textbf{HTTP 304} ante un
\texttt{If-None-Match} coincidente, lo que habilita \emph{caching} agresivo en
navegador y proxies (RNF-05).

El valor de esta pieza para la reusabilidad es directo: un script externo
puede pedir el \emph{bundle} una sola vez y generar interfaces TypeScript,
formularios precompilados, validadores o documentación; y un microservicio
puede espejar el contrato de otro sin acoplarse a su ORM.

\subsubsection{Polimorfismo y relaciones del ORM}
El cierre del incremento añadió las dos piezas que faltaban, ambas inyectadas
en \texttt{get\_enriched\_json\_schema} para que afecten por igual a
\texttt{/<entity>/schema.json} y al \emph{bundle} agregado. La primera es el
\textbf{polimorfismo}: si la clase ORM es la raíz de una jerarquía
(\texttt{polymorphic\_on} definido y sin mapper padre), el esquema emite un
\texttt{oneOf} con el esquema enriquecido de cada subclase y un
\texttt{x-discriminator} (\texttt{propertyName} +
\texttt{mapping} valor$\rightarrow$subclase) deducido de
\texttt{object\_type\_id}. La segunda es el reflejo de los
\texttt{relationship()} del ORM como un array \texttt{x-relationships}
(nombre, destino, dirección, si es colección, columnas FK locales y
\emph{endpoint} si la entidad está registrada), que complementa el
\texttt{x-foreign-key} por columna ya existente.

\subsubsection{Decisiones de diseño y problemas encontrados}
La decisión más relevante fue \textbf{incrustar las variantes completas} en el
\texttt{oneOf} en lugar de referenciarlas con \texttt{\$ref} a
\texttt{\#/\$defs}: dentro del \emph{bundle} las entidades cuelgan de
\texttt{entities.<nombre>}, por lo que un \texttt{\#/\$defs} no resolvería
desde la raíz del documento; incrustarlas mantiene el \emph{bundle}
autocontenido. La segunda fue \textbf{conservar las \texttt{properties} base}
junto al \texttt{oneOf}: así un consumidor que aún no entienda \texttt{oneOf}
---como el conversor a Formly actual--- sigue viendo las columnas comunes sin
romperse, lo que permitió acotar el alcance en el frontend a no desestabilizar
el conversor (sin un selector de tipo todavía). La tercera fue el
\textbf{determinismo}: las variantes se ordenan por valor de discriminador y
las relaciones por nombre, de modo que el \texttt{version}/ETag del
\emph{bundle} permanece estable entre generaciones idénticas, requisito de
\emph{caching} (RNF-05). Un problema colateral surgido al ejecutar la batería
de pruebas en la base SQLite en memoria fue que el tipo PostgreSQL
\texttt{TSVECTOR} no es compilable en SQLite; se resolvió, en línea con los
tipos ya independientes de plataforma (\texttt{GUID}, \texttt{JSONB}), con un
\texttt{@compiles} que lo representa como \texttt{TEXT} fuera de PostgreSQL,
sin alterar el comportamiento nativo.

\section{Pruebas}
Cada incremento se acompañó de pruebas automatizadas, en coherencia con el
requisito de fiabilidad (RNF-04).

\subsection{Backend}
Las pruebas del backend se ejecutan con \texttt{pytest}. Destacan:
\texttt{test\_envelope.py} (forma del envoltorio, 404, validación,
\texttt{/sys/entities}, \texttt{schema.json}), \texttt{test\_auth.py}
(autenticación), \texttt{test\_form\_schema.py} (presencia de
\texttt{x-ui-widget} en columnas \emph{datetime} y \texttt{x-options-endpoint}
en \emph{foreign keys}, más \emph{unit tests} de
\texttt{\_ui\_hints\_for\_column}) y \texttt{test\_schema\_bundle.py}
(versión estable, \emph{round-trip} de ETag con 304, sensibilidad del hash al
contenido).

\subsection{Frontend}
En el frontend, \texttt{schema-to-formly.spec.ts} cubre los tipos primitivos,
\texttt{required}, \emph{enum}/\emph{select}, los \emph{overrides}
\texttt{x-formly-type} y \texttt{x-ui-widget}, \texttt{x-options-endpoint},
\texttt{fieldGroup}, \texttt{fieldArray}, resolución de \texttt{\$ref}, las
expresiones (\texttt{x-hide-when}) y el modo \texttt{view}, incluyendo el caso
de una raíz polimórfica con \texttt{oneOf}/\texttt{x-discriminator}. El flujo
extremo a extremo --- descubrimiento, obtención de esquema, generación del
formulario y envío mediante \texttt{EntityClient}, mostrando las incidencias
del envoltorio --- queda encapsulado en el componente reutilizable
\texttt{<ngt-dynamic-form>}, que compone esas piezas sin acoplarse al dominio.

puntos donde un cambio del modelo de datos podría romper la cadena de
generación, que es exactamente el riesgo que el framework pretende
neutralizar.
